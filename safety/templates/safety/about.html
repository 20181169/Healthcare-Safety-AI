왜 단일 YOLO 만으로는 부족한가

공사현장 CCTV·스마트폰 사진은 (a) 새벽/야간/터널 등 저조도, (b) 원거리/모바일 카메라의 저해상도, (c) 작업자 움직임으로 인한 모션 블러 가 결합되어 있습니다. 단일 단계 YOLO 만으로는 confidence 가 떨어져 미탐(false negative) 이 늘고, 이를 막으려 임계값을 낮추면 오탐(false positive) 이 폭증하는 trade-off 에서 빠져나올 수 없습니다.

STACK

기술 스택 선택 근거

YOLOv5s (Ultralytics)
1차 탐지 백본
6 MB · ~10 ms/frame@416

v8 대비 v5s 를 선택한 이유: (1) 현장 GPU(엣지 장비) 호환성, (2) 가중치 다운로드 안정성, (3) 416×416 입력에서 RT 시연 가능. mAP 는 v8 이 약간 우위지만 latency 5배 차이.

Zero-DCE++
저조도 보정 (조건부)
52 KB · ~3 ms/frame

GAN-based 저조도 보정(EnlightenGAN 등) 과 달리 unsupervised + 초경량. 52 KB 라는 사이즈는 엣지 배포의 결정적 이점. 또한 노출 보정 곡선(curve estimation) 방식이라 색감 왜곡이 적음.

Real-ESRGAN ×4 (ROI 한정)
머리 영역 SR
67 MB · ROI 만 ~17 ms

전체 이미지 SR 은 100 ms+ 로 실시간 불가. YOLO 가 찾아준 머리 ROI 만 잘라 SR 하면 17 ms 로 처리 가능. ROI < 64 px 인 경우에만 트리거 → 깨끗한 이미지는 그대로 통과.

Head ROI 이중분류기 (presence + correct)
YOLO 후단 보정

YOLO 의 NO-Hardhat 클래스 단독으론 false alarm 가능성 — 머리 ROI 를 잘라 {착용 여부, 올바른 착용 여부} 두 분류기를 통과시켜 최종 판정. 단계별 confidence 가 모두 임계값을 넘을 때만 위반 alarm 송출. (주: 보존된 모델은 2-class — 이중분류기 학습은 별도 진행)

조건부 적용 게이트 ★
대회 차별화 포인트

Zero-DCE/SR 를 모든 프레임에 무조건 적용하면 깨끗한 이미지의 latency 만 늘어남. 품질 측정(밝기/대비/노이즈) → 임계값 위반 시에만 모듈 활성화 라는 게이트를 두어 평균 latency 를 baseline 과 동일 수준 유지하면서 어려운 케이스에서만 성능 향상.

💡 종합 결론

Zero-DCE 의 helmet detection 효과는 환경 조명에 강하게 의존. 정상 조명 (밝기 ≥ 115) 에선 효과 미미하지만, 매우 어두운 환경 (밝기 ≤ 66) 에선 AP +0.0754p (상대 +14.3%), F1 +0.053 의 명확한 회복. 시스템 설계 가정인 야간/터널 공사현장에서의 가치는 정량적으로 검증됨. 단 현재 게이트 임계값 (밝기 60) 은 정상 조명용 — 야간 운영 시 더 관대한 임계값 (예: 밝기 100) 또는 무조건 적용이 더 좋을 가능성.

ARCHITECTURE

파이프라인 (7 단계)

1. 품질 측정
brightness · contrast · noise
2. Zero-DCE++ (조건부)
밝기 < 70 일 때 활성
3. YOLOv5s 탐지
Hardhat · NO-Hardhat (실측 2-class)
4. Real-ESRGAN (조건부)
ROI < 64px 일 때 활성
5. Head 이중분류
presence + correct
6. 융합 판정
OK / NO_HELMET / INCORRECT
7. 얼굴 비식별
블러/픽셀화

핵심: 2번·4번 단계는 1번·3번 게이트의 결과에 따라 조건부로만 실행됨. 정상 sample 에서는 baseline 과 동일 latency, 어려운 sample 에서만 추가 비용이 발생하는 적응형 파이프라인.

회고 / 한계

  • • 모델 클래스 차이: 설계 의도 4-class (person/helmet/no_helmet/head) 였으나 보존된 checkpoint 는 2-class (Hardhat/NO-Hardhat). 학습 시 통합·축소된 것으로 추정 — 이중분류기/ROI 게이트 등 후단 단계는 설계 의도대로 작동하지만 정량 평가는 helmet 한 축만 가능.
  • • noise 게이트는 결함: 밝은 sample 에 저조도 보정 적용되어 손실 발생. 데이터 분석으로 정량 진단 후 제거.
  • • 합성 darkening 의 한계: 카메라 underexposure 시뮬레이션 (감마 보정) 으로 진짜 야간 sample 대체 — 진짜 터널·새벽 데이터셋에서 재검증 필요.
  • • 게이트 임계값 운영 환경별 분기 필요: 정상 조명 (밝기 60) vs 야간 운영 (밝기 100+) 별도 설정 — 현 구현은 단일 임계값.
  • • Real-ESRGAN 가중치를 잘못 로드하면 YOLO 입력 형식이 깨져 탐지 0 으로 무너지는 케이스 확인 — runtime guard 필요.
  • • 실시간 스트림: 백그라운드 추론 스레드 + 박스 TTL 0.8s 로 영상 부드러움과 박스 정확도 균형.
